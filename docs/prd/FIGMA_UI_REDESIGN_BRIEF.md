# NexusClaw Figma UI 重设计 Brief（V1）

## 1. 目标

- 在不改变核心业务流程的前提下，重做门户端与后台端视觉和交互层。
- 统一「政务可信 + 高效操作」的产品气质，提升首屏理解效率与任务完成率。
- 建立可复用设计系统（颜色、字体、间距、组件、状态）以支撑后续迭代。

---

## 2. 设计范围

### 2.1 门户端（portal-web）

- 登录
- 知识问答
- 知识检索（若与问答同屏，定义清晰切换与反馈模式）
- 文档问答
- 历史会话
- 个人中心

### 2.2 后台端（admin-web）

- 登录、工作台
- 系统配置：开头语、Prompt
- 知识管理：知识库、知识条目、知识树、文件库
- 测试管理：测试集、测试任务、任务详情
- 日志查询：日志列表、日志详情
- 用户管理：用户列表、用户编辑
- 项目管理：项目列表、新建、编辑、成员管理
- 体验广场：知识问答、知识检索、文档问答

---

## 3. 设计原则

- 信息优先：先让用户看懂「当前项目、当前能力、当前状态」。
- 操作闭环：每个关键页面必须具备「入口、执行、反馈、回退」。
- 状态完整：loading、空状态、错误、权限不足、成功反馈必须成套定义。
- 一致性：同类表格、筛选、弹窗、抽屉、步骤流保持统一交互语义。
- 可实现：组件命名与状态要与前端实现可一一映射。

---

## 4. 建议视觉方向（政务场景）

- 主色：蓝青系（可信、清晰），避免高饱和炫色干扰信息密度。
- 强调色：橙/琥珀用于提醒和风险，不与主操作色冲突。
- 字体层级：标题稳重、正文高可读，减少花哨装饰字体。
- 布局：后台偏高密度；门户偏引导与可读性。
- 动效：只在切换、加载、反馈时使用短动效，避免装饰性动画。

---

## 5. 信息架构与页面优先级

### P0（先做，直接影响主链路）

- 门户：知识问答、文档问答、历史会话
- 后台：项目管理、知识库/文件库、Prompt 配置、日志查询

### P1（第二批）

- 后台：知识树编辑、测试管理全链路、用户管理
- 门户：帮助与功能推荐页

### P2（收口优化）

- 全局状态页、权限提示页、异常反馈页统一
- 组件细节与响应式适配优化

---

## 6. 核心组件清单（设计系统最小集）

- 导航：顶部栏、侧边栏、面包屑、项目切换器
- 输入：Input、Textarea、Select、DatePicker、SearchBar
- 操作：Button（主/次/危险/幽灵）、IconButton、Dropdown
- 数据：Table、Tag、Badge、Pagination、Tabs、Steps
- 反馈：Toast、Alert、Modal、Drawer、Empty、Skeleton
- 业务：聊天气泡、来源知识卡、知识状态标签、日志详情面板

---

## 7. 状态与规则（设计必须覆盖）

- 权限：超级管理员 / 项目管理员 / 项目成员差异化可见与可操作状态
- 知识状态：编辑中、上线中、生效中、上线失败、下线中、下线失败
- 文件状态：已上传、解析中、解析成功、解析失败、切片中、切片成功、切片失败
- 测试任务状态：未运行、运行中、成功、失败、已取消
- 知识树版本状态：编辑中、已发布、已归档

---

## 8. Figma 文件组织建议

- Page 1：`00_Foundations`（色板、字体、栅格、间距、阴影、圆角）
- Page 2：`01_Components`（基础组件 + 业务组件 + 变体）
- Page 3：`02_Portal_Screens`（门户全流程页面）
- Page 4：`03_Admin_Screens`（后台全流程页面）
- Page 5：`04_Prototype`（关键流程原型）
- Page 6：`05_Handoff`（标注、交互规则、异常说明）

---

## 9. 交付标准（DoD）

- 每个页面至少包含：默认态、空态、加载态、异常态、权限态（如适用）。
- 所有组件有命名规范、变体规范、尺寸规范、状态规范。
- P0 页面具备可点击原型，能完整演示主业务链路。
- 关键页面与当前前端路由具备一一映射关系，便于开发落地。

---

## 10. 下一步执行建议

1. 先完成 `00_Foundations` 与 `01_Components`（避免后续页面返工）。
2. 并行出两条主链路原型：
   - 门户：登录 -> 问答 -> 来源查看 -> 历史会话
   - 后台：项目管理 -> 知识/文件管理 -> 问答 -> 日志回溯
3. 第一轮评审只看「信息结构与任务路径」，第二轮再看视觉精修。
