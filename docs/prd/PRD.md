## 2. 产品定位

Nexus RAG 平台 V2.0 是一套面向企业与政务场景的知识问答与知识管理平台，支持用户通过知识库、文件库、知识树、知识编译层、项目配置与底层检索能力，快速搭建可落地的智能问答系统。

在当前产品演进口径中，平台的知识链路不再只定义为“一次提问、一次临时检索、一次生成回答”，而是采用三层模式：

- 原始资料层：政策文件、FAQ、网页、表格、会议纪要、案例材料等事实依据。
- 知识编译层：将散乱资料和优质问答沉淀为结构化知识页，维护交叉引用、版本与健康状态。
- 问答与办理层：面向用户提供咨询、补问、材料清单、流程说明与办理推进。

其中，知识编译层用于提升长期知识资产质量，不替代原始资料，也不替代底层检索与最终问答。

平台同时服务两类核心场景：

- C 端 / 门户体验场景：提供知识问答、知识检索、文档问答等体验能力。
- B 端 / 管理后台场景：提供项目、知识、文件、知识树、测试、日志、用户与权限等管理能力。

## 3. 产品目标

### 3.1 业务目标

- 帮助业务团队快速搭建可控、可配置、可追溯的问答系统。
- 降低知识整理、维护、测试与上线成本。
- 将文件、知识、知识树、Prompt 与问答体验统一在一个平台中完成闭环。
- 将高频问答、对比分析和政策解释沉淀为长期复用的知识资产，而不是停留在单次会话中。
- 支撑政务、企业知识咨询、业务办理、客服问答等复杂场景。

### 3.2 产品目标

- 支持项目级知识问答、多轮对话和答案生成。
- 支持知识检索与检索结果辅助问答。
- 支持文档问答、文档预览、文件切分入库和 QA 自动生成。
- 支持开头语、推荐问题、Prompt、项目能力等系统配置。
- 支持知识库、知识树、文件库的统一管理。
- 支持知识编译、交叉引用、知识健康检查与答案回流沉淀。
- 支持测试集、测试任务和自动评测。
- 支持历史对话日志查询、专家评测与问答回流。
- 支持用户管理、项目管理和多级权限控制。

## 4. 目标用户与权限角色

### 4.1 目标用户

#### 超级管理员

- 管理平台全部项目与全部用户。
- 配置系统权限、项目权限与底层能力。
- 负责平台级治理与运维。

#### 项目管理员

- 管理自己参与项目的配置、知识、文件、知识树、成员与日志。
- 可编辑项目配置与项目成员。
- 不可管理系统级用户账号。

#### 项目成员

- 主要使用体验广场、测试管理、日志查询等功能。
- 可查看系统配置，但不具备项目与用户管理权限。

#### 体验用户

- 在门户中使用知识问答、知识检索、文档问答等功能。
- 关注回答质量、知识来源与使用体验。

### 4.2 权限层级

- 系统角色：`super_admin`、`normal_user`
- 项目角色：`project_admin`、`project_member`

### 4.3 权限规则

- 超级管理员拥有全平台读写能力。
- 项目管理员仅可管理自己参与的项目，且每个项目最多配置 5 位项目管理员。
- 项目成员不可访问用户管理、项目管理，不可编辑关键系统配置。
- 门户体验用户只访问前台体验能力，不直接暴露后台管理能力。

## 7. 功能分解

## 7.1 访问门户

### 7.1.1 功能推荐

- 目的：帮助用户快速理解平台能力与可用入口。
- 入口：门户首页。
- 主要动作：查看能力介绍、跳转体验模块、查看示例。
- 输出结果：引导用户进入知识问答、知识检索、文档问答。
- 空状态：无项目或无功能时展示默认引导。
- 错误状态：功能配置加载失败时展示重试。

### 7.1.2 使用帮助

- 目的：帮助用户快速上手各功能模块。
- 入口：门户帮助入口。
- 主要动作：查看功能说明、操作说明和常见问题。
- 输出结果：提升首访上手效率。

### 7.1.3 项目选择

- 目的：让用户基于部门、业务、知识范围选择要使用的项目。
- 入口：门户首页、登录后切换项目。
- 主要动作：按部门/业务筛选、选择项目。
- 输出结果：切换当前项目上下文。
- 依赖：项目列表、项目权限。

### 7.1.4 个人中心

- 目的：维护当前用户个人资料。
- 主要动作：修改昵称、密码、用户描述。
- 输出结果：用户信息更新。

## 7.2 体验广场

### 7.2.1 知识问答

- 目的：基于项目知识、项目能力和大模型生成答案。
- 目标用户：体验用户、项目成员、项目管理员、超级管理员。
- 入口：体验广场默认页。
- 主要动作：
  - 新建对话
  - 多轮问答
  - 打开高级功能设置
  - 切换敏感检测/搜索相关性过滤/知识树等开关
  - 选择生效知识库
  - 查看答案来源知识
  - 专家评测答案
- 输出：
  - 模型答案
  - 来源知识 TopN
  - 引用文献
  - 对话历史
- 回答策略：
  - 优先读取已整理知识页与知识树上下文
  - 若证据不足，再回查原始知识条目、文件切片与检索结果
- 依赖：
  - 项目配置
  - 知识编译层
  - 生效知识库
  - 检索能力
  - 大模型推理能力
- 空状态：展示开头语、推荐问题与新对话引导。
- 错误状态：模型超时、知识不足、敏感拦截、服务异常。

### 7.2.2 知识检索

- 目的：从生效知识中返回相关知识与辅助答案。
- 入口：体验广场 -> 知识检索。
- 主要动作：
  - 输入问题搜索
  - 查看 Top10 检索结果
  - 查看标题、标签、内容、更新时间、来源文件
  - 对搜索结果人工评测
  - 让系统基于结果生成问答答案
- 输出：
  - Top10 检索结果
  - 搜索问答助手答案
  - 来源知识与参考文档
- 空状态：显示检索说明与默认示例。
- 错误状态：搜索失败、无结果、结果不可用。

### 7.2.3 文档问答

- 目的：基于指定文档进行问答与原文定位。
- 入口：体验广场 -> 文档问答。
- 主要动作：
  - 上传文件
  - 选择是否覆盖同名文件
  - 选择已有文档
  - 预览原文
  - 对当前文档提问
- 输出：
  - 文档列表
  - 文档预览
  - 文档问答记录
  - 原文高亮定位
- 依赖：文件上传、文档解析、文档索引。
- 错误状态：上传失败、解析失败、问答失败、预览失败。

## 7.3 系统配置

### 7.3.1 开头语配置

- 目的：配置门户开场文案与推荐问题展示形式。
- 模式：
  - 文本模式
  - 卡片模式
- 支持内容：
  - 开头语文本
  - 推荐问题
  - 热门问题
  - 热门政策
  - 是否生效
- 结果：影响 C 端首页与问答页首屏展示。

### 7.3.2 Prompt 配置

- 目的：配置项目级 Prompt 与模板变量。
- 支持内容：
  - Prompt 正文
  - 模板变量 `{qa}`、`{history}`、`{query}`
  - 版本管理与提交生效
- 结果：影响问答生成策略。

## 7.4 知识管理

### 7.4.1 知识库

- 目的：管理结构化知识条目。
- 主要动作：
  - 新建知识
  - 编辑知识
  - 批量导入 Excel
  - 上线/下线知识
  - 查询知识
  - 添加标签/关键词
  - 自动生成相似问题
- 重要字段：
  - 文档名称
  - 标题
  - 关键词
  - 内容
  - 状态
  - 来源
- 看板指标：
  - 全部知识
  - 编辑中
  - 上线中
  - 生效中
  - 上线失败
  - 下线中
  - 下线失败
- 业务规则：
  - 建议单条知识不超过 4000 字
  - 生效知识才参与检索与问答

### 7.4.2 知识树

- 目的：用树状结构承载复杂业务引导与多轮反问。
- 主要动作：
  - 拖拽编辑树
  - 添加多级节点
  - 编辑节点名称、信息、权重、条件列表、知识列表
  - 自动规整
  - 重置
  - 更新版本
  - 历史版本管理
  - 下载与发布
  - 上传树 JSON
- 节点类型：
  - 中间节点
  - 叶子节点
- 业务规则：
  - 叶子节点可配置条件列表
  - 节点可关联知识
  - 同级节点权重影响反问逻辑
  - 仅发布版本对线上生效

### 7.4.3 文件库

- 目的：统一管理上传文档及文档衍生能力。
- 主要动作：
  - 上传文件
  - 同名覆盖
  - 预览
  - 下载
  - 删除
  - 生成 QA
  - 切分入库
- 支持文件类型：
  - `pdf`
  - `doc`
  - `docx`
- 上传约束：
  - 单文件不超过 100M
  - 单次最多 30 个文件
- 切分策略：
  - 按长度切分
  - 按目录切分
- 结果：文件可转化为知识条目进入知识库。

### 7.4.4 知识编译层

- 目的：将原始资料、已生效知识和高质量问答整理为长期维护的结构化知识页，降低重复综合成本，提升知识复用率与治理能力。
- 主要动作：
  - 摄入新资料并归类
  - 自动生成或更新知识页
  - 建立页面之间的交叉引用
  - 将优质问答、对比分析和业务说明回流为知识页
  - 执行知识健康检查
  - 维护知识目录与变更日志
- 关键产物：
  - 知识页集合
  - `index` 目录
  - `log` 变更记录
  - 健康检查结果
  - 页面到原始资料的来源映射
- 业务规则：
  - 知识页属于派生资产，不替代原始依据
  - 事实性结论必须可追溯到原始资料、文件或知识条目
  - 发现矛盾、过时、孤立页面或缺失概念时，应进入治理与复核流程
  - 问答层可优先读取知识页，但高风险结论仍需支持回查原始来源

## 8. 核心业务规则

### 8.1 项目隔离

- 所有知识、文件、知识树、测试数据、日志均按项目隔离。
- 门户侧问答只能使用当前选定项目下的可用知识与配置。

### 8.2 知识生效规则

- 只有生效中的知识参与知识检索与知识问答。
- 文档上传后的内容需经过解析、切分或生成 QA 后才进入知识库。

### 8.3 知识树生效规则

- 只有已发布的知识树版本对线上会话生效。
- 编辑中的知识树不影响线上版本。

### 8.4 Prompt 生效规则

- 项目级 Prompt 为问答主模板。
- 对话中若切换功能开关，只影响当前会话，不自动覆盖项目默认配置。

### 8.5 评测规则

- 不同测试任务类型对数据集字段有不同要求。
- 缺少 `ref_answer` 的数据集不得用于依赖参考答案的评测任务。

### 8.6 日志与审计规则

- 问答链路关键过程需留痕。
- 管理后台关键增删改操作需进入操作日志。

### 8.7 知识编译规则

- 原始资料层是事实依据的最终来源，知识编译层是可复用表达层。
- 知识编译层可沉淀优质答案、对比分析和结构化说明，但不得脱离来源独立生效。
- 编译后的知识页需保留更新时间、来源范围与版本痕迹。
- 对外回答若依赖编译页，仍需支持回查原始文档、原始知识条目或原始文件片段。

## 10. 非功能要求

### 10.1 性能

- 常规查询接口应支持后台分页与筛选。
- 检索返回应在可接受时延内完成。
- 大模型调用需支持超时控制与失败提示。

### 10.2 可观测性

- 保留问答链路日志、检索结果、Prompt 快照与用户反馈。
- 后台关键操作要有操作日志。

### 10.3 安全

- 后台需登录鉴权。
- 按系统角色与项目角色双层控制权限。
- 敏感能力与模型供应商密钥需走服务端配置。

### 10.4 可维护性

- 产品口径、接口、表结构、页面路由必须统一。
- 新增功能需同步更新本 PRD、API 与 DB 文档。
- 知识编译页、来源映射、健康检查结果与变更日志需可持续维护与审计。

## 11. 验收标准

### 11.1 核心验收

- 用户可以登录后台并进入有权限的项目。
- 门户可完成知识问答、知识检索、文档问答。
- 后台可完成开头语、Prompt、知识、知识树、文件、测试、日志、用户、项目的管理闭环。
- 文件可上传、预览、切分入库并产生可检索知识。
- 平台可将部分原始资料编译为知识页，并在问答中优先复用已整理内容。
- 测试集与测试任务可执行并产出结果。
- 日志可回看改写结果、Prompt 与引用来源。

### 11.2 失败场景验收

- 无权限用户不可访问受限页面与接口。
- 无生效知识时系统需返回明确提示而非编造答案。
- 文档解析失败、模型失败、检索失败需有明确状态与提示。

### 11.3 发布约束

- 上线前须保证 PRD、技术文档、API 文档、数据库文档与页面 IA 一致。
- 若缺少原型设计，至少需先完成页面清单、状态设计与交互文案规范。
