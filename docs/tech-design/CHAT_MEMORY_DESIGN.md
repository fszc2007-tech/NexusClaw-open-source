# NexusClaw 会话记忆与上下文设计方案

## 1. 目标

本方案用于约束 NexusClaw 在市民服务场景中的对话记忆设计，目标是让前台问答具备连续上下文理解能力，同时避免把“记忆”做成后台运营对象或事实依据来源。

适用范围：

- `portal-web` 门户问答
- `api-server` 中的 chat orchestration
- 项目级配置、审计与删除能力

不适用范围：

- 后台显式运营记忆内容
- 将记忆作为政策结论依据
- 长期保存敏感个人信息

---

## 2. 设计原则

### 2.1 前台有记忆感，后台弱显式

- 前台应让用户感受到“能接上上文”
- 后台不展示记忆详情运营台
- 后台仅提供开关、审计、删除能力

### 2.2 记忆只做上下文理解

- 记忆用于 query 改写、槽位补全、答案组织
- 政策事实、材料要求、资格判断、办理时限必须来自知识库、规则库或明确来源
- 当记忆与知识来源冲突时，以知识来源为准

### 2.3 敏感信息默认不做长期沉淀

- 身份证号、手机号、住址、详细个人经历默认不得写入长期记忆
- 敏感信息最多在当前会话短时使用
- 会话结束或超过保留期后应自动清理

### 2.4 结构化优先

- 优先记“办事状态”而不是原始聊天文本
- 优先记录低风险偏好而不是个人身份细节
- 优先用结构化字段驱动下游链路

### 2.5 可审计、可删除、可回退

- 每次问答需记录是否使用了记忆
- 记忆提取结果应可在日志中追踪
- 会话记忆必须支持删除和清空

---

## 3. 三层记忆模型

## 3.1 会话短期记忆

用途：

- 解决代词、省略、连续追问
- 维持当前会话的自然衔接

数据组成：

- 最近 N 轮对话
- 当前会话摘要 `summary`
- 最近一次有效检索主题

示例：

```json
{
  "recent_turns": [
    {
      "query": "港澳通行证怎么办",
      "answer": "首次办理需准备身份证、照片回执等材料"
    },
    {
      "query": "那要预约吗",
      "answer": "部分地区支持预约办理"
    }
  ],
  "summary": "用户正在咨询港澳通行证首次办理，当前关注材料和预约。"
}
```

## 3.2 任务状态记忆

这是市民服务场景的核心记忆层。

用途：

- 将“用户在办什么事”抽成结构化状态
- 在追问时补齐隐含上下文
- 为检索和回答生成提供明确约束

推荐字段：

```json
{
  "intent": "办理港澳通行证",
  "region": "香港",
  "user_profile": {
    "identity_type": "首次办理",
    "applicant_type": "本人"
  },
  "current_focus": [
    "材料",
    "流程",
    "预约"
  ],
  "missing_slots": [
    "办理地区"
  ]
}
```

设计要求：

- 任务状态优先写入 `chat_sessions.state_json`
- 每轮对话后增量更新
- 当状态缺失且影响事实判断时，优先追问澄清，不得直接猜测

## 3.3 轻量长期偏好记忆

用途：

- 优化表达方式，不改变事实依据

允许范围：

- 常用语言
- 回答风格偏好
- 是否偏好简洁答复
- 常咨询业务类型

禁止范围：

- 身份证号
- 手机号
- 住址
- 银行卡
- 家庭详细情况
- 未经明确同意的身份属性

示例：

```json
{
  "language": "zh-Hans",
  "answer_style": "brief",
  "preferred_topics": [
    "出入境",
    "社保"
  ]
}
```

## 4. 系统架构

## 4.1 记忆在链路中的角色

```text
用户问题
-> 读取项目配置
-> 读取会话短期记忆
-> 读取任务状态记忆
-> 读取低风险长期偏好
-> 生成结构化上下文
-> query 改写 / 槽位补全
-> 检索知识库 / 规则库
-> 基于来源生成答案
-> 更新摘要与任务状态
-> 写入日志与审计
```

### 4.2 事实源分层

- 记忆层：提供上下文，不提供事实结论
- 知识库层：提供政策、材料、流程、口径
- 规则层：提供资格、时效、地区限制、硬性约束

回答必须同时满足：

- 能说明当前理解到的上下文
- 能指出事实来源来自知识或规则
- 无来源时宁可说明不确定，也不从记忆推断结论

### 4.3 后台边界

后台不建设“记忆内容管理台”，仅提供以下治理能力：

- 项目级记忆开关
- 项目级保留时长设置
- 会话级记忆清空
- 日志中查看本轮是否使用记忆
- 审计删除动作

---

## 5. 数据设计

## 5.1 项目级配置

建议在 `project_settings` 中增加：

- `capability_memory`：是否启用记忆能力
- `memory_scope`：`off` / `session_only` / `session_and_preference`
- `memory_ttl_days`：会话记忆保留天数
- `preference_memory_enabled`：是否启用长期偏好记忆

说明：

- `session_only` 为默认推荐值
- 政务场景首版建议默认关闭长期偏好记忆，按项目灰度开启

## 5.2 会话级数据

建议在 `chat_sessions` 中使用：

- `summary`：自然语言摘要
- `state_json`：结构化任务状态

推荐 `state_json` 结构：

```json
{
  "intent": "办理港澳通行证",
  "region": "香港",
  "user_profile": {
    "identity_type": "首次办理"
  },
  "current_focus": [
    "材料",
    "预约"
  ],
  "missing_slots": [
    "办理城市"
  ],
  "updated_at": "2026-04-05T10:30:00Z"
}
```

## 5.3 消息级审计数据

建议在 `chat_messages` 中增加：

- `used_memory`：本轮是否使用记忆
- `memory_snapshot_json`：本轮实际拼装进 Prompt 的记忆快照
- `safety_flags_json`：敏感信息检测与脱敏标记

用途：

- 支持审计和问题定位
- 便于评估记忆是否提升回答质量

## 5.4 长期偏好存储

建议新增 `user_preference_memories`：

- 仅存低风险偏好
- 按 `project_id + user_id` 隔离
- 支持禁用和删除

推荐结构：

```json
{
  "preferences_json": {
    "language": "zh-Hans",
    "answer_style": "brief"
  },
  "source": "explicit_confirmed",
  "status": "active"
}
```

## 6. 问答链路设计

## 6.1 上下文组装顺序

每次 `chat/ask` 按以下顺序组装上下文：

1. 项目 Prompt 与项目默认能力配置
2. 当前会话最近 N 轮对话
3. 会话摘要 `summary`
4. 会话任务状态 `state_json`
5. 低风险长期偏好
6. 用户当前问题

优先级规则：

- 当前问题 > 当前会话状态 > 会话摘要 > 长期偏好

## 6.2 Query 改写规则

当满足以下条件时允许用记忆改写 query：

- 用户问题存在代词或省略
- 当前问题是上一轮话题的自然追问
- 任务状态中的槽位足以补全 query

不得做的事：

- 不得把敏感信息拼进改写后的 query
- 不得把用户历史偏好改写为事实判断
- 不得在缺关键槽位时直接假设地区、资格或人群属性

## 6.3 答案生成规则

回答生成时应遵循：

- 先依据检索到的知识和规则回答
- 再根据任务状态调整表达顺序
- 当上下文不足时主动追问
- 当知识来源不足时说明“不确定”或“需补充地区/身份信息”

推荐回答模板：

```text
基于当前会话信息，您咨询的是 {intent}。
根据检索到的知识，{answer_from_sources}
如您办理地区为 {missing_or_known_region}，建议进一步确认 {slot}
```

## 6.4 记忆更新规则

每轮回答完成后：

1. 生成新的会话摘要
2. 更新任务状态
3. 检测并剔除敏感信息
4. 如满足条件，再更新长期偏好
5. 写入本轮审计字段

长期偏好更新前提：

- 信息属于低风险偏好
- 有明确表达或足够高置信度
- 未与现有偏好冲突

---

## 7. 安全与合规

## 7.1 敏感信息策略

默认视为敏感的信息包括：

- 证件号
- 手机号
- 联系地址
- 详细出生日期
- 银行账户

处理策略：

- 会话内可短暂使用，但默认不入长期偏好
- 进入日志前应做脱敏标记
- 超过保留期后自动清理

## 7.2 删除与清理

系统需支持：

- 清空单个会话的 `summary` 与 `state_json`
- 删除长期偏好记录
- 保留删除动作审计日志

建议清理任务：

- 每日扫描过期会话记忆
- 清空过期摘要、状态和敏感快照

## 7.3 风险控制

当满足以下任一条件时，禁止依赖记忆补全并应主动澄清：

- 涉及资格判断
- 涉及地区政策差异
- 涉及年龄、人群、身份特殊限制
- 当前状态置信度不足

---

## 8. Portal 体验设计

前台应呈现“连续理解”的体验，但避免过强的“我记住你了”的表达。

推荐做法：

- 自动续接最近会话
- 在会话列表中展示上次咨询标题
- 在必要时轻提示“已结合本次会话上下文”

不推荐做法：

- 显式展示完整记忆档案
- 在回答中频繁强调“我记得你之前说过”
- 暴露结构化任务状态给普通用户

---

## 9. Admin 治理设计

后台仅保留治理能力：

- 记忆开关
- 保留天数
- 是否允许长期偏好记忆
- 会话记忆清空
- 审计查询

不提供：

- 记忆内容批量编辑
- 偏好画像运营
- 记忆推荐运营

---

## 10. MVP 实施建议

## 10.1 第一阶段

- 在 `chat_sessions` 启用 `summary`
- 新增 `state_json`
- 在问答链路中使用最近 N 轮对话 + `summary` + `state_json`
- 审计本轮是否使用记忆

## 10.2 第二阶段

- 引入敏感信息识别与脱敏
- 增加项目级记忆开关和保留时长
- 增加会话记忆清空能力

## 10.3 第三阶段

- 新增长期偏好表
- 灰度启用低风险偏好记忆
- 做记忆效果评估与 A/B 对比

---

## 11. 结论

NexusClaw 的记忆设计应以“会话连续性 + 办事状态理解”为中心，而不是构建显式的记忆运营系统。首版最有价值、风险最低的做法，是优先落地会话短期记忆与任务状态记忆，把长期偏好记忆作为灰度能力。
