# Nexus RAG 平台 V2.0 技术基线

## 1. 文档说明

本技术方案是仓库统一技术基线，来源于以下资料的合并收敛：

1. Nexus 问答知识库系统技术方案
2. Nexus RAG 平台 V2.0 功能清单
3. 智谱大模型问答系统使用手册

本文件用于约束系统边界、模块职责、数据流、部署与安全设计。若现有代码实现与本方案冲突，以本方案为目标基线进行重构。

补充专题文档：

1. `RAG_CAPABILITY_AUDIT.md`：当前 RAG 能力盘点、开源复用建议与轻量改造路线

---

## 2. 设计目标

- 支持项目级智能问答平台的完整后台与门户闭环。
- 支持知识、文件、知识树、Prompt、测试与日志的统一治理。
- 支持知识编译、知识页沉淀和知识健康检查，形成长期可复用知识资产。
- 支持多轮问答、检索增强问答、文档问答三条主链路。
- 支持面向市民服务场景的会话记忆与任务状态理解。
- 支持按项目隔离的权限、数据与配置。
- 在架构上兼容政务与企业知识场景，支持后续扩展。

## 3. 技术选型

## 3.1 前端

- React
- Umi / Umi Max
- TypeScript
- Ant Design

前端分为两端：

- `portal-web`：门户体验端
- `admin-web`：后台管理端

## 3.2 后端

- Python 3.11+
- FastAPI
- Uvicorn
- SQLAlchemy
- Alembic

## 3.3 数据与基础设施

- MySQL：业务主数据库
- Redis：缓存、短期会话、任务状态
- Elasticsearch：term-based 检索
- FAISS：向量检索
- Nginx：统一入口代理与负载均衡

## 3.4 模型与算法

- 智谱 GLM-4：主问答与自动评测
- ChatGLM-Pro / ChatGLM-Turbo：文档 QA 生成、评测与问答任务
- BGE Embedding：知识向量化
- BGE Reranker：召回结果重排
- TEI：Embedding / 小模型推理服务

```text
Infra:
- MySQL
- Redis
- Elasticsearch

- Object Storage / local file storage

```

补充说明：

- 在原始资料与问答层之间，新增建议建设 `knowledge compilation service`，也可称为知识编译层或知识整理层。
- 它的职责不是替代 `retrieval service`，而是把散乱资料、高质量答案和多文档综合结果沉淀为长期维护的结构化知识页。
- 问答链路应优先查询知识编译页；当知识页证据不足或问题高度时效化、个案化时，再回查原始知识条目与检索结果。

### 4.1 形态建议

首版建议采用“模块化单体 + 可拆分服务边界”的方式落地：

- 对外暴露统一的 `api-server`
- 内部先以模块形式组织能力
- 文件解析、向量化、评测任务可异步化
- 后续根据压力拆出 retrieval / evaluation / document 独立服务

这样既能快速交付，又不会把后续拆分路径堵死。

## 5. 模块职责

## 5.1 api-server

职责：

- 登录鉴权
- 权限校验
- 项目上下文识别
- 请求路由聚合
- 统一响应格式
- 审计埋点

不负责：

- 复杂业务决策
- 检索与模型编排细节

## 5.2 project / config 模块

职责：

- 项目 CRUD
- 项目成员管理
- 项目能力配置
- 开头语配置
- Prompt 配置
- 项目默认知识库与默认行为配置

## 5.3 knowledge service

职责：

- 知识库 CRUD
- 知识条目 CRUD
- 标签/关键词维护
- Excel 导入
- 上线/下线状态流转
- 相似问题管理

## 5.4 knowledge tree service

职责：

- 知识树版本管理
- 节点编辑
- 条件列表与权重维护
- 节点关联知识
- 发布/回滚/下载/上传

## 5.5 document service

职责：

- 文件上传与存储
- 原文预览
- 下载与删除
- 解析文本
- 生成 QA
- 切分入库
- 向知识编译层提供可消费的原始资料与结构化解析结果

## 5.6 retrieval service

职责：

- term-based 检索
- 向量检索
- 多路召回
- 蛇形 merge / 综合 merge
- 多级标题合并
- 多分域相关性
- 动态权重融合重排
- 可用性校验

## 5.7 chat orchestration

职责：

- 读取项目配置
- 读取会话历史
- 读取会话摘要与任务状态记忆
- 多轮 query 改写
- 优先读取知识编译页与知识页索引
- 调 retrieval service
- 组装 Prompt
- 调用 GLM 生成回答
- 返回来源知识
- 写入会话与日志
- 更新会话摘要、状态与审计字段

## 5.8 dedup service

职责：

- 知识内容标准化
- 相似知识召回
- 重复分值计算
- 输出候选与原因
- 记录处理动作

## 5.9 evaluation service

职责：

- 管理测试数据集
- 调度测试任务
- 执行机跑答案
- 调用模型进行自动评测
- 汇总评测结果

## 5.10 knowledge compilation service

职责：

- 接收文件解析结果、已发布知识条目和高质量问答
- 生成或更新结构化知识页
- 维护页面分类、交叉引用与目录索引
- 记录知识编译变更日志
- 执行知识健康检查，例如矛盾、过时、孤立页面和缺失概念识别
- 为问答层提供“优先可读、可复用”的知识表达层

不负责：

- 替代原始资料存储
- 直接替代底层 term / vector 检索
- 在无来源依据时独立产出事实结论

## 6. 核心数据流

## 6.1 知识问答链路

```text
用户问题
-> 鉴权 / 获取项目上下文
-> 读取项目默认配置与会话开关
-> 读取已选知识库与会话历史
-> 读取会话摘要 / 任务状态 / 低风险偏好
-> query 改写与槽位补全
-> 查询知识编译页索引 / 已整理知识页
-> 判断知识编译页证据是否足够
-> 若证据不足则执行 term 检索 + 向量检索
-> merge + rerank
-> 检索结果可用性校验
-> 组装 Prompt
-> 调用 GLM 生成答案
-> 返回答案 + 来源知识
-> 更新摘要与状态
-> 写入 chat_sessions / chat_messages / logs
```

## 6.2 知识检索链路

```text
输入问题
-> 项目校验
-> 查询生效知识
-> term 检索 + 向量检索
-> 融合排序
-> 返回 Top10 结果
-> 可选触发“基于结果生成答案”
```

## 6.3 文档问答链路

```text
上传文档 / 选择已有文档
-> 文件存储
-> 文档解析
-> 建立文档片段索引
-> 用户提问
-> 仅在指定文档范围内检索
-> 生成答案
-> 返回答案 + 原文高亮定位
```

## 6.4 文件切分入库链路

```text
上传文件
-> 解析文本
-> 选择切分策略
-> 生成知识候选
-> 写入知识条目
-> 可选触发知识编译
-> 进入知识状态流转
```

## 6.5 自动评测链路

```text
选择测试集 + 任务类型
-> 调用问答/检索链路生成机跑结果
-> 调用 GLM 做自动评测
-> 产出任务汇总与明细
-> 支持人工复核
```

## 6.6 知识编译链路

```text
新资料进入 / 高质量问答回流
-> 解析并识别主题、概念、来源范围
-> 创建或更新知识页
-> 建立交叉引用和目录索引
-> 写入编译日志
-> 执行 lint / 健康检查
-> 输出可供问答层优先读取的知识页
```

## 7.3 后端职责

## 8. 第三方与供应商边界

## 8.1 模型供应商

- 问答与评测：智谱 GLM 系列
- QA 生成：ChatGLM-Pro / ChatGLM-Turbo

要求：

- 服务端必须显式校验密钥存在性。
- 未配置模型密钥时必须返回明确错误，不能 silently fallback 到假数据。

## 8.2 检索供应商

- Elasticsearch：term 检索
- FAISS：向量召回

要求：

- 检索引擎不可用时，应返回“检索服务不可用”错误。
- 不可直接用静态 mock 结果替代正式召回。

## 8.3 文件与存储

- 文件内容需支持预览、下载、切分与入库。
- 建议抽象存储适配层，兼容本地磁盘与对象存储。

## 9. 数据设计原则

- 一切业务数据按项目隔离。
- 生效状态与编辑状态分离。
- 记忆只用于上下文理解，不作为政策结论依据。
- 原始资料层与知识编译层必须分离，前者是事实依据，后者是派生知识资产。
- 知识编译页必须保留来源映射、更新时间与变更痕迹。
- 敏感信息默认不进入长期记忆。
- 重要链路留痕：
  - 对话日志
  - Prompt 快照
  - 检索来源
  - 记忆使用情况与脱敏标记
  - 操作日志
  - 测试结果
- 发布态与编辑态隔离：
  - 知识
  - 知识树
  - Prompt / 开头语

## 9.1 会话记忆设计

市民服务场景下的记忆设计采用三层模型：

- 会话短期记忆：最近 N 轮对话与会话摘要
- 任务状态记忆：事项、地区、用户身份、当前关注点等结构化状态
- 轻量长期偏好：语言、回答风格、常咨询业务类型等低风险偏好

边界要求：

- 记忆仅参与 query 改写、槽位补全和回答组织
- 政策、流程、材料、资格等事实结论必须来自知识库或规则层
- 后台不做记忆内容运营，仅做开关、审计和删除

详细方案见：

- `docs/tech-design/CHAT_MEMORY_DESIGN.md`

## 10. 权限设计

## 10.1 认证

- 后台所有接口需登录后访问。
- 门户体验能力可根据项目策略开放匿名或登录态使用。

## 10.2 授权

- 系统角色控制平台级能力。
- 项目角色控制项目级能力。
- 任何接口都需同时校验：
  - 用户是否已登录
  - 是否拥有目标项目访问权
  - 是否拥有当前动作的权限

## 10.3 页面可见性

- 超级管理员：全部页面可见。
- 项目管理员：无用户管理编辑权、无非参与项目访问权。
- 项目成员：不可见用户管理与项目管理。

## 10.4 知识编译层权限映射

- `super_admin`
  - 可管理全部项目的编译知识页、编译运行、健康检查与回流合并
- `project_admin`
  - 可管理所属项目的编译知识页、编译运行、健康检查与回流合并
- `project_member`
  - 默认只读，可查看编译知识页、来源映射、版本与健康结果
  - 不可发布页面、发起编译运行、合并回流候选

说明：

- 如后续引入更细粒度 `kb read / kb write / kb manage / kb admin`，应作为现有系统角色与项目角色的派生权限层，而不是替换当前权限体系。

## 11. 状态与任务设计

## 11.1 同步接口

适用于：

- 查询列表
- 查询详情
- 普通配置保存
- 轻量级检索

## 11.2 异步任务

适用于：

- 文件解析
- QA 生成
- 切分入库
- 知识编译
- 知识健康检查
- 批量导入
- 自动评测

任务要求：

- 可查询状态
- 可失败重试
- 保留执行日志
- 需支持幂等键
- 同一编译页同一时间仅允许一个运行中的 `recompile` / `backfill`
- 编译失败不得覆盖当前发布版本
- 重复提交相同幂等键时，应返回已有任务而不是重复创建

## 15. 当前仓库整改结论

当前仓库中的 `portal-web`、`admin-web`、`api-server` 可以保留为载体，但需要按本方案进行回收与补完：

- `portal-web`：从简化聊天页升级为完整体验广场。
- `admin-web`：从缺页骨架升级为完整管理后台。
- `api-server`：从 mock/占位接口升级为真实业务 API。

本文件即后续整改与开发的目标架构基线。
